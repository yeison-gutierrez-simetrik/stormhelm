# 20 — Agents

**Scope.** Formal Claude Code sub-agents that Stormhelm defines as part of the framework. Distinct from the main agent (Claude Code itself) and from Ralph workers / Agent Teams teammates (which are operational patterns without persistent definitions). An agent here is a versioned, named role with a constrained tool set, designed to be invoked when the main agent cannot do the job well by itself.

**When to read.** Considering whether to formalize a new role as an agent vs. a skill; reviewing why an agent has a specific tool set; integrating an agent into a workflow step.

**Rules in this file.** §114, §115, §116

> See `../AGENTS.md` for the full rule index. Related: `12-bdd-and-acceptance.md` (the gates the reviewer audits), `13-ralph-and-afk.md` (Ralph + Agent Teams use the reviewer), `17-bug-handling.md` (postmortems are the future `postmortem-writer` agent's domain), `16-security-supply-chain.md` (the future `security-auditor` agent's domain).

---

## Why agents are scarce in Stormhelm

The default in Stormhelm is **the main agent + skills + rules + hooks**, not multi-agent. A formal sub-agent is only justified when **all of**:

1. The job is recurring (not a one-off).
2. The main agent does it worse because of **confirmation bias** (it just wrote the code being audited), **context saturation** (the relevant rules and the in-flight work would not both fit cleanly in the main session), or **tool-set mismatch** (the role needs strictly fewer tools than the main agent has, and that constraint is the point).
3. No existing skill already covers it.
4. The setup overhead is paid back by repeated invocations.

Most candidates fail one of those. The framework has **one** shipped agent today (`reviewer`) and **two specified, deferred** agents (§115, §116) to be implemented when their triggers arrive.

## §114. Independent code review is mandatory before any draft PR opens; the reviewer agent runs read-only and cites `§N` violations explicitly

Before any PR opens — whether generated by the main agent during interactive work, by `/feature` skill at Step 12, by a Ralph worker after `/tdd` Green + `/run-acceptance` pass, or by an Agent Teams teammate — the **reviewer agent** must audit the diff and produce a structured findings report.

The reviewer is read-only by design. Its tool set is `Read, Grep, Glob, Bash` and the Bash matcher only allows non-mutating commands (`git diff`, `git log`, `git show`, `gh pr view`, `gh pr diff`, `git blame`). Any other Bash command is the agent's signal to stop and report instead.

### Why a separate agent and not just a skill

The main agent that wrote the code has **confirmation bias**: it spent tokens defending the design as it was implemented. Asking the same agent to audit its own work yields predictable false negatives. The reviewer is invoked **fresh**, with no context about why the code looks the way it does — and that distance is the value.

This is the same reason `git blame` is more useful than `git annotate-with-author-explanations`: the author's reasoning is exactly what we want to ignore at review time.

### Required output

The reviewer produces a structured Markdown report with sections:

- **Header**: PR / diff target, file count, line count, rules loaded, timestamp.
- **🛑 Blocking findings**: each with `§N` violated, file:line, one-sentence issue, fix direction (in prose, not code).
- **⚠️ Should fix**: same shape, lower severity.
- **💡 Suggestions**: optional improvements.
- **What the author got right**: mandatory section that names concrete strengths.
- **Summary table**: counts per severity + recommendation (`do not merge` / `merge after blocking resolved` / `merge after author addresses Should fix` / `approve as-is`).

The "what the author got right" section is not decorative. It calibrates trust so criticisms cannot be dismissed as reflexive.

### Severity discipline

| Severity | Threshold |
|---|---|
| 🛑 Blocking | Rule violation that compromises architecture, security, auditability, or the public contract |
| ⚠️ Should fix | Style or quality violation that author can document a reason for keeping |
| 💡 Suggestion | Preference, not violation |

Inflating ⚠️ → 🛑 destroys the gate's credibility (boy-who-cried-wolf). Downgrading 🛑 → ⚠️ ships bad code. The reviewer agent's system prompt explicitly enforces this calibration.

### Integration points

| Invoker | When |
|---|---|
| `/code-review` skill | Direct human or main-agent invocation |
| `/feature` skill Step 12 | Automatic, before `gh pr create --draft` |
| `ralph-local.sh` | Automatic, after `/tdd` Green + `/run-acceptance` pass |
| Agent Teams §107 | The `reviewer` teammate role at Tasks 8, 9 (QA per module) is this agent |

### Bypass — only with constitution amendment

Skipping the reviewer for a specific PR requires writing an explicit ADR in `docs/adr/` documenting why this PR is exempt (typically: trivial change to documentation only). The bypass is logged in the PR description. No silent skips.

### Failure mode handling

If the reviewer cannot complete the review (diff empty, branch conflicted, critical rule files missing), it **escalates** to the main agent with a structured message — never produces a partial review pretending to be complete.

### Why this is §114 and not a skill

A skill is invoked in the **same agent session** with the **same accumulated context**. An agent is a **separate session** with a **separate context**. For code review, the second is necessary. Formalizing it as `§114 + agents/reviewer.md` makes the architectural choice explicit and auditable.

### Where the agent definition lives

The canonical file lives at `agents/reviewer.md` in the Stormhelm repository (alongside `hooks/` and `skills/`). On `/setup`, it is copied or symlinked to `.claude/agents/reviewer.md` in the consuming project — the path Claude Code looks for sub-agents.

Frontmatter:

```yaml
---
name: reviewer
description: Independent code review agent. Activates AFTER code is written...
tools: Read, Grep, Glob, Bash
---
```

Plus the system prompt body (~3 pages of behavior rules). See `agents/reviewer.md` for the canonical text.

## §115 (specified, deferred). Postmortem-writer agent for production incidents

When the team accumulates 3+ production incidents with structured Ralph session logs (§69) and TEMPLATE.md-based postmortems (§95), the cost of writing postmortems by hand is repeated and high-value. At that point, formalize a `postmortem-writer` agent.

### Specification

```yaml
---
name: postmortem-writer
description: Generates postmortem documents from incident artifacts. Reads Ralph
  session logs, git history, PR descriptions, and the active /run-acceptance
  output around the time of the incident; produces a draft following
  .claude/skills/postmortem/references/postmortem-template.md. The human reviewer always finalizes and signs
  off — this agent produces the draft, never the official document.
tools: Read, Grep, Glob, Bash
---
```

### Why an agent and not a skill

- **Context freshness**: a postmortem written immediately after the incident by the same agent that participated in the response inherits the response's blind spots. A fresh agent reading the artifacts catches what the responders missed.
- **Tool set constraint**: only needs Read/Grep/Glob/Bash. Cannot — and should not — modify code, run tests, or deploy. The constraint is the point.
- **Recurring**: once production has incidents at any volume, postmortems are weekly work.

### Trigger to implement

- 3+ postmortems written by hand using `.claude/skills/postmortem/references/postmortem-template.md` in the last quarter, **or**
- The team explicitly requests it during a retrospective.

### Anti-trigger (do not implement before)

- Fewer than 3 incidents — the per-incident customization burden exceeds the benefit.

## §116 (specified, deferred). Security-auditor agent for compliance-driven hardening

When the project comes under SOC2, ISO 27001, EU AI Act, GDPR, HIPAA, or any compliance framework that requires documented periodic security reviews, formalize a `security-auditor` agent.

### Specification

```yaml
---
name: security-auditor
description: Conducts proactive security audits following STRIDE methodology
  (§87, §101). Reads the codebase, threat models in docs/threat-models/, recent
  PRs touching auth/crypto/PII, and produces a structured audit report citing
  §84-§90, OWASP Top 10, and the relevant compliance framework controls. Read-
  only; finds and reports, does not fix.
tools: Read, Grep, Glob, Bash
---
```

### Why an agent and not a skill

- **Context freshness**: the main agent that wrote the auth code is the worst auditor of it.
- **Tool set constraint**: read-only matches the auditor's mandate.
- **Recurring**: SOC2 and similar frameworks require periodic audits with documented evidence — at least quarterly, often more.

### Trigger to implement

- Project under active compliance program (SOC2, ISO 27001, EU AI Act, GDPR, HIPAA), **or**
- First external penetration test (§90) returns findings that should have been caught internally.

### Anti-trigger (do not implement before)

- No compliance requirement. The reviewer agent (§114) reading `core/16-security-supply-chain.md` is sufficient for most security review until a regulator says otherwise.

## What about all the roles in §107 (Agent Teams)?

The roles mentioned in §107 (`arch-{mod}`, `fe-{mod}`, `devops`, `reviewer`, `integrator`, `qa-final`) are **operational patterns**, not formal agents. They are spawn-with-prompt structures inside a parallelization session, not versioned tool-set-constrained agents.

The single exception is the `reviewer` teammate in §107, which **is** this `agents/reviewer.md` agent — invoked from inside an Agent Team for per-module QA (Tasks 8 and 9 in the §107 task graph).

## Why not more agents

Three considerations keep this list short:

1. **Each agent is operational surface area to maintain.** Frontmatter, system prompt, tool set, integration points, tests. The reviewer is justified; the others wait.
2. **Skills compose better than agents.** A skill runs in the user's session; an agent opens a separate one. Skills are cheaper to call in sequence.
3. **The rule set + main agent + skills covers most roles by reading the right rules at the right moment.** The pattern "agent reads `core/16-security-supply-chain.md` and behaves as a security auditor" is real and effective for ad-hoc work — it just doesn't survive the bias problems when the audit must happen on code the same agent wrote.

When in doubt, **prefer a skill over an agent**. Agents are the exception.
